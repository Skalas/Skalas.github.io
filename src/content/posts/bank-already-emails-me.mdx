My bank emails me every time I swipe a card. Amount, merchant, date, all of it, pushed to my inbox in real time without my asking. For years I treated those as noise to archive. They're the only API I need.

Personal finance tools sell you a false choice. Either you connect your real bank accounts to a third party — Plaid, Yodlee, whatever Copilot runs on — and hand your full transaction history to a SaaS, or you go back to filling a spreadsheet every Sunday night like it's 2005. Automation in exchange for your data, or privacy in exchange for your weekends.

You don't have to pick. The bank is already pushing the data to you, one email per transaction. Parse it locally and the dilemma disappears.

In the last post I said the finance tracker earned one of its own. Here it is.

## The shape

Three small pieces, no daemon, no cloud call:

```
[Bank email] → (Himalaya + Keychain) → [Python parser] → [SQLite: finance.db]
                                                               │
                                                               ▼
[cron] ───────────────────────────────────────────────→ [SQL reports]
```

Each piece does one thing. Nothing runs in the background eating RAM, and nothing phones home.

## The bank is the API

BAC Credomatic sends an email the moment a card is charged. To read those without putting a mailbox password in a config file in my repo, I use [Himalaya](https://github.com/pimalaya/himalaya), a CLI mail client that talks IMAP.

Himalaya pulls the password from the macOS Keychain on demand — `security find-generic-password` under the hood — so the secret lives in memory for the length of the run and nowhere else. The filter runs server-side against `All Mail`, not the inbox:

```
from:baccredomatic.com after:yesterday
```

<Aside>
  Filtering on `All Mail` instead of the inbox was a real fix, not a preference. Gmail's labeling rules sometimes move the bank alert out of the inbox before the script runs, and a transaction that isn't in the inbox is a transaction that silently never gets logged. Search everything, lose nothing.
</Aside>

## Parsing emails that have accents

Latin American bank alerts are messier than they look. HTML tables laid out sideways, and merchant names full of accents and `ñ`. A naive regex truncates "Farmacia Económica" or chokes on "Almacén González," and you find out weeks later when a category is quietly wrong.

So the patterns cover the full Latin set explicitly — `[A-Za-z0-9áéíóúÁÉÍÓÚñÑüÜ\s\.\*\-\&]` — and the line that pulls a transaction out of the body looks like this:

```python
pattern = r"Fecha\s+(\d{2}/\d{2}/\d{2})\s+Moneda\s+([^\s]+)\s+Monto\s+([0-9,]+\.[0-9]{2})\s+(.+?)\s+Estado"
```

## SQLite, not CSV

The first version wrote transactions to a CSV. CSV is a bad database pretending to be a file. No atomicity, no real concurrency, and if the process dies halfway through a write you get a corrupted line and a bad afternoon. Sorting and deduping all happened in Python, in memory, every run.

I moved the whole thing to a local SQLite file. One `finance.db`, a real schema, and the database handles what I was hand-rolling:

```sql
CREATE TABLE IF NOT EXISTS transactions (
    msg_id        TEXT PRIMARY KEY,   -- email message id
    date          TEXT NOT NULL,      -- YYYY-MM-DD
    time          TEXT NOT NULL,      -- HH:MM
    merchant      TEXT NOT NULL,
    amount        REAL NOT NULL,
    currency      TEXT DEFAULT 'USD',
    category      TEXT DEFAULT 'Uncategorized',
    status        TEXT DEFAULT 'Auto',
    source        TEXT DEFAULT 'Email',
    original_text TEXT                 -- raw snippet, for audits
);
```

## Killing duplicates with a hash

The email message id is a fine primary key when every transaction comes from an email. But some don't — a manual deposit, an old row imported from one of those legacy CSVs. Those have no message id, so they'd slip past the key and double up.

For those I build the key myself: a SHA-256 over the fields that can't change — date, time, merchant, amount. Same transaction, same hash, every time. Then `INSERT OR IGNORE` lets SQLite drop the duplicate at write time, so I never write a read-compare-write loop in Python to catch it.

## Reports are just SQL

This is the payoff for using a real database. A monthly breakdown by category is one query — no pandas, no juggling lists in memory:

```sql
SELECT
    category,
    ROUND(SUM(amount), 2) AS total,
    COUNT(*)              AS n
FROM transactions
WHERE date LIKE '2026-06%'
GROUP BY category
ORDER BY total DESC;
```

cron runs the parser, the parser feeds the database, and the database answers questions. The whole pass finishes in under two seconds, and not a byte of it leaves the disk.

## Build your own

The bank-specific part is small: the regex that reads one bank's email format. Everything downstream — the schema, the dedup, the reports — is the same no matter who you bank with. Split those two and the parser becomes the only thing anyone has to swap to point this at their own bank.

If you want to start, the order that worked for me was: read your own bank alert and write the regex first, point it at a throwaway `finance.db`, then wire up cron once the numbers look right. Your transaction history is worth more than the convenience you'd trade it for.
